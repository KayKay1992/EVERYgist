Delete Blog post functionality

==========================================
HOW I IMPLEMENTED DELETE BLOG POST FUNCTIONALITY
==========================================

OVERVIEW:
Implemented a complete delete workflow for blog posts with confirmation modal, backend API endpoint, and UI feedback. The flow includes: clicking delete button → opening confirmation modal → confirming deletion → API call → success notification → refreshing the list.

==========================================
STEP 1: BACKEND - DELETE ENDPOINT
==========================================

File: backend/controllers/blogPostController.js

Created a controller function to handle blog post deletion:

```javascript
// @desc    Delete a blog post
// @route   DELETE /api/posts/:id
// @access  Admin
const deletePost = async (req, res) => {
  try {
    const post = await BlogPost.findById(req.params.id);
    if (!post) return res.status(404).json({ message: "Post not found" });
    await post.deleteOne();
    res.status(200).json({ message: "Post deleted successfully" });
  } catch (err) {
    res.status(500).json({ message: "Server Error", error: err.message });
  }
};
```

Key Features:
- Checks if post exists before deletion (404 if not found)
- Uses `deleteOne()` to remove from database
- Returns success message on completion
- Handles errors with 500 status

File: backend/routes/blogPostRoutes.js

Route Configuration:
```javascript
router.delete("/:id", protect, adminOnly, deletePost);
```

- Protected route (requires authentication)
- Admin-only access (prevents unauthorized deletions)
- Uses post ID from URL parameter

A) STATE FOR DELETE MODAL:
```javascript
const [openDeleteAlert, setOpenDeleteAlert] = useState({
  open: false,
  data: null,
});
```

- `open`: Boolean to control modal visibility
- `data`: Stores the post ID to be deleted
- Initialized as closed with no data

B) DELETE FUNCTION:
```javascript
const deletePost = async (postId) => {
  try {
    await axiosInstance.delete(API_PATHS.POSTS.DELETE(postId));
    toast.success("Blog Post Deleted Successfully");
    setOpenDeleteAlert({ open: false, data: null });
    getAllPosts(); // Refresh the list
  } catch (error) {
    console.error("Error deleting post:", error);
  }
};
```

Function Breakdown:
1. **API Call**: Uses Axios DELETE request with post ID
2. **Success Toast**: Shows success notification to user
3. **Close Modal**: Resets modal state to closed
4. **Refresh List**: Fetches updated blog posts list
5. **Error Handling**: Logs errors to console

STEP 4: MODAL COMPONENT

Reusable modal component with the following features:

```javascript
const Modal = ({ children, isOpen, onClose, hideHeader, title }) => {
  if (!isOpen) return null; // Don't render if closed
  
  return (
    <div className="fixed inset-0 z-50 flex items-center justify-center w-full h-full bg-black opacity-90">
      <div className="relative flex flex-col bg-white shadow-lg rounded-lg overflow-hidden">
        {/* Header with title */}
        {!hideHeader && (
          <div className="flex items-center justify-between p-4 border-b border-gray-200">
            <h3 className="md:text-lg font-medium text-gray-900">{title}</h3>
          </div>
        )}
        
        {/* Close button (X) */}
        <button type="button" className="..." onClick={onClose}>
          <svg>...</svg>
        </button>
        
        {/* Modal body (scrollable) */}
        <div className="flex-1 overflow-y-auto custsom-scrollbar">
          {children}
        </div>
      </div>
    </div>
  );
};
```

Key Features:
- **Fixed Overlay**: Dark background covering entire screen
- **Centered Content**: Modal positioned in center
- **Close Button**: X button in top-right corner
- **Optional Header**: Can hide header if needed
- **Scrollable Body**: Handles long content
- **Z-index**: Ensures modal appears above other content
- **Conditional Rendering**: Returns null when closed (optimization)

==========================================
KEY FEATURES & DESIGN DECISIONS
==========================================

1. CONFIRMATION MODAL:
   - Prevents accidental deletions
   - Clear, explicit user action required
   - Standard UX pattern for destructive actions

2. IMMEDIATE FEEDBACK:
   - Success toast notification
   - Modal closes automatically
   - List refreshes to show updated state

3. ERROR HANDLING:
   - Backend validates post exists (404 if not)
   - Frontend logs errors to console
   - Could be enhanced with error toast

4. STATE MANAGEMENT:
   - Single state object for modal (open + data)
   - Clean state resets after action
   - No lingering state issues

5. REUSABLE COMPONENTS:
   - Modal works for any content
   - DeleteAlertContent works for any delete action
   - Can be used throughout the app

6. SECURITY:
   - Admin-only route protection
   - Authentication required
   - Server-side validation

==========================================
ERROR SCENARIOS HANDLED
==========================================

1. **Post Not Found** (404):
   - Backend checks if post exists
   - Returns 404 if already deleted
   - Frontend logs error

2. **Unauthorized** (403):
   - Admin-only middleware catches this
   - Non-admin users blocked at API level

3. **Server Error** (500):
   - Database errors caught
   - Error message returned
   - Frontend logs to console

4. **Network Error**:
   - Axios catches network failures
   - Error logged to console
   - User sees no feedback (could be improved)

==========================================
TECHNICAL DECISIONS EXPLAINED
==========================================

1. **WHY USE MODAL FOR CONFIRMATION?**
   - Forces user attention
   - Blocks other interactions
   - Clear, focused action
   - Industry standard for destructive actions

2. **WHY REFRESH ENTIRE LIST?**
   - Ensures data consistency
   - Simple implementation
   - Gets updated counts for tabs
   - Could be optimized to remove single item

3. **WHY STORE POST ID IN STATE?**
   - Delays API call until confirmation
   - Allows modal to be generic (reusable)
   - Separates UI state from data operations
   - Easier to test and debug

4. **WHY CLOSE MODAL BEFORE REFRESH?**
   - Better UX (immediate visual feedback)
   - Prevents user from clicking multiple times
   - Shows action completed
   - Standard pattern

5. **WHY USE TOAST NOTIFICATION?**
   - Non-intrusive feedback
   - Doesn't block UI
   - Auto-dismisses
   - Standard pattern users expect
