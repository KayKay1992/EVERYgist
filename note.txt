## Login Component Implementation Explanation

File: `frontend/gist_hub/src/components/Auth/Login.jsx`

Purpose
- Provide a controlled login form that authenticates a user, stores the JWT token, updates global auth state via `UserContext`, and routes admins to their dashboard.

Key Pieces
1. React State
	- `email`, `password`: controlled inputs.
	- `error`: holds validation or API error messages.
2. Context
	- `updateUser`, `setOpenAuthForm` from `UserContext` to store user data and close any auth modal/sheet.
3. Navigation
	- `useNavigate()` from React Router to redirect after successful login (admin → `/admin/dashboard`).
4. Validation
	- `validateEmail(email)` helper ensures proper email format.
	- Checks for missing password.
5. API Call
	- Uses shared `axiosInstance` and `API_PATHS.AUTH.LOGIN` to POST `{ email, password }`.
	- Extracts `token` and `role` from response.
6. Auth Persistence
	- Saves `token` to `localStorage` immediately (`localStorage.setItem('token', token)`).
	- Calls `updateUser(response.data)` to populate global user state (so other components can see `user` without re-fetching).
7. Conditional Redirect
	- If `role === 'admin'`, navigates to `/admin/dashboard`.
	- Non-admin simply closes auth form (could later redirect to a landing/home page).
8. Error Handling
	- If Axios returns a structured error (`error.response.data.message`), display it.
	- Otherwise, show a generic fallback message.
9. Switching to SignUp
	- A button toggles parent state by calling `setCurrentPage('signup')` to swap the view to the SignUp component.
10. UX Enhancements
	- Shows an error message inline above the submit button.
	- Includes a side image (`AUTH_IMG`) for larger screens (hidden on small viewports).

Data Flow (Step-by-Step)
1. User enters email/password → local component state updates.
2. User presses LOGIN → `handleLogin` runs `e.preventDefault()`.
3. Validate email and password; if invalid, set `error` and abort.
4. Clear previous `error` and send POST request.
5. Backend authenticates and responds with `{ token, role, ...user }`.
6. Store token in `localStorage`, call `updateUser` (context now has user + token).
7. Close auth form (`setOpenAuthForm(false)`).
8. If admin, navigate to `/admin/dashboard`.
9. On failure → capture message and show under form.

Files Involved
- `Login.jsx` (this component)
- `UserContext` (`userContext.jsx`) for global state
- `axiosInstance` for standardized HTTP config
- `apiPaths.js` for endpoint references
- `helper.js` for `validateEmail`
- `Input.jsx` for reusable form input component
- Asset: `AUTH_IMG` for visual enhancement

Why This Approach
- Separation of concerns: validation + API + context update each in clear blocks.
- Reusable building blocks (`Input`, `axiosInstance`, `API_PATHS`) minimize duplication.
- Immediate token persistence allows subsequent protected requests without another login cycle.

Possible Improvements
- Add a loading spinner state to disable the submit button during the request.
- Use an Axios interceptor to automatically inject the token instead of directly setting it (already likely in `axiosInstance`).
- Sanitize or trim input before submission.
- Add rate limiting / lockout UI after multiple failed attempts.
- Display password requirements or a “Forgot password?” link.

Edge Cases Considered
- Invalid email format → early validation prevents unnecessary API call.
- Empty password → stops before request.
- API error (network vs server validation) → gracefully handled with specific vs generic message.
