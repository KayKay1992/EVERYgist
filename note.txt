CORE FUNCTIONALITY:
-------------------

A) HEADING EXTRACTION (useEffect #1)
   Purpose: Parse markdown content and extract all headings
   
   Process:
   1. Regex Pattern: /^(#{1,6)\s+(.+)$/gm
      - ^ : Start of line
      - (#{1,6}) : Capture 1-6 hash symbols (heading level)
      - \s+ : One or more whitespace
      - (.+) : Capture heading text
      - $ : End of line
      - gm : Global, multiline flags

   2. Loop through all matches using regex.exec()
   
   3. For each match:
      - Extract level: match[1].length (count # symbols)
      - Extract text: match[2].trim()
      - Generate ID:
        * Convert to lowercase
        * Remove special characters: /[^\w\s-]/g
        * Replace spaces with hyphens: /\s+/g
        * Example: "Getting Started with React" â†’ "getting-started-with-react"
   
   4. Store in headings array: [{ level, text, id }, ...]
   
   Dependencies: [content]
   Triggers: When blog post content changes


B) SCROLL TRACKING (useEffect #2)
   Purpose: Detect which section user is currently reading
   
   Process:
   1. Add scroll event listener to window
   
   2. On scroll event (handleScroll):
      a) Map through all headings
      b) Find corresponding DOM element by ID
      c) Get bounding rect (position relative to viewport)
      d) Store { id, top } for each heading
      e) Filter out null values
   
   3. Find active heading:
      - Look for heading where top > 0 AND top < 200
      - 200px threshold = early highlighting for better UX
      - If none found, use last visible heading (scrolled past)
   
   4. Update activeId state
   
   5. Cleanup: Remove event listener on unmount
   
   Dependencies: [headings]
   Triggers: User scrolls page


C) SMOOTH SCROLL FUNCTION
   Purpose: Navigate to clicked heading smoothly
   
   Function: scrollToHeading(id)
   
   Process:
   1. Find DOM element by ID
   2. Get element position: getBoundingClientRect().top
   3. Calculate scroll position with offset:
      - Offset: 100px (account for sticky navbar)
      - Target: elementPosition + pageYOffset - offset
   4. Scroll with smooth behavior:
      window.scrollTo({ top: target, behavior: "smooth" })
   
   Why offset? Prevents heading from hiding behind navbar


D) CONDITIONAL RENDERING
   Purpose: Only show TOC when useful
   
   Logic:
   if (!headings || headings.length < 2) return null;
   
   Rationale:
   - 0-1 headings = No navigation needed
   - 2+ headings = TOC provides value
   - Keeps UI clean for short posts

SCROLL DETECTION ALGORITHM:
----------------------------

Goal: Determine which section is "active" (currently being read)

Approach:
1. Get all heading positions relative to viewport
2. Find heading within "active zone" (0-200px from top)
3. Fallback: If user scrolled past all, use last heading

Why 200px threshold?
- Better UX: Highlights section slightly before reaching top
- Gives user context of what's coming
- Feels more responsive
- Accounts for reading position (eyes are middle of screen)

Edge cases handled:
- No headings in viewport â†’ Keep last active
- First heading not yet reached â†’ No active state
- User jumps to bottom â†’ Last heading stays active


SMOOTH SCROLL IMPLEMENTATION:
------------------------------

Function: scrollToHeading(id)

Why offset needed?
- Navbar: Sticky at top (height ~80px)
- Without offset: Heading hides behind navbar
- With 100px offset: Perfect positioning

Calculation:
1. Element position from top of page: element.getBoundingClientRect().top
2. Current scroll position: window.pageYOffset
3. Target position: elementPosition + pageYOffset - 100

window.scrollTo parameters:
- top: Target scroll position
- behavior: "smooth" (animated scroll, ~500ms duration)

Alternative: behavior: "auto" (instant jump)
Chosen: "smooth" for better UX

EDGE CASES HANDLED
------------------

1. No Headings:
   - Component returns null
   - No empty UI rendered
   - Sidebar space used by other content

2. Single Heading:
   - Component returns null
   - TOC not useful for single section
   - Keeps UI clean

3. Very Long Heading Text:
   - line-clamp-2: Truncates to 2 lines
   - Prevents layout breaking
   - Shows ellipsis (...)

4. Headings with Special Characters:
   - Regex removes: !@#$%^&*()+=[]{}|;:'",.<>?/\
   - Keeps: Letters, numbers, hyphens
   - Ensures valid HTML IDs

5. Duplicate Heading Text:
   - Current: Both get same ID (not ideal)
   - Future fix: Add index suffix (e.g., "heading-1", "heading-2")
   - Low priority: Rare in real content

6. User Scrolls Very Fast:
   - Scroll event throttling: Not implemented
   - Native browser throttles scroll events
   - No performance issues observed

7. Content Changes During Scroll:

   - Smooth transition


DESIGN DECISIONS EXPLAINED
---------------------------

Q: Why 2+ headings minimum?
A: Single heading = no navigation needed. TOC adds value for multi-section articles only.

Q: Why sticky positioning?
A: Keep navigation always accessible. Users can jump to any section without scrolling back up.

Q: Why collapse feature?
A: Long articles may have 10+ sections. Collapsing reduces visual clutter while keeping access nearby.

Q: Why show progress bar?
A: Gamification element. Encourages reading completion. Helps users estimate remaining time.

Q: Why 200px active threshold?
A: Early highlighting feels more responsive. Shows what's coming, not just what's past.

Q: Why hierarchical indentation?
A: Visual representation of content structure. H2 under H1, H3 under H2, etc.

Q: Why smooth scroll?
A: Instant jumps are jarring. Smooth scroll shows path to destination. Better spatial awareness.

Q: Why purple/pink theme?
A: Consistency with app design. Matches buttons, gradients, and brand colors throughout.
